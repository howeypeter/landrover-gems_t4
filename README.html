gems_t4

A TestBook-T4-style diagnostic & (emulated) programming tool for the Lucas/SAGEM GEMS V8 engine ECU — P38 Range Rover 4.0/4.6 & Discovery 1 V8i (1995–early 1999).

Python 3.11+ Virtual ECU — no car needed K-line / KWP2000 153 tests passing

It speaks the GEMS K-line (KWP2000 / ISO-14230) dialect and ships with a built-in virtual ECU, so the whole thing runs on a laptop with no car and no adapter. When you're ready for real hardware, the same code drives a Raspberry Pi Pico K-line adapter over USB.

Scope note. GEMS engine maps live in socketed UV-EPROMs and are not reflashable over the K-line — that capability never existed for GEMS. This tool does the real over-the-wire work (fault codes, live data, actuator tests, coding, immobiliser Security-Learn) and treats map “reflash” as an emulated chip-swap lookalike. Background dossier is in memory/research/.

New to the P38/GEMS platform? See How the electronics work — a visual walkthrough of the K-line diagnostic star, the BeCM hub, and how the ECU, EAT, ABS, EAS, and body systems talk to each other.

Requirements

  • Python 3.11 or newer (developed on 3.14).
  • Windows, macOS, or Linux for the CLI. (The GUI is developed and packaged on Windows.)
  • No hardware required for --fake mode (the default).

Install

Create a virtual environment and install the dependencies. Activate the venv once per terminal — after that, plain python and pip point at it, so there are no path prefixes to type.

Windows (PowerShell)

python -m venv .venv
.venv\Scripts\Activate.ps1
pip install -r requirements.txt

macOS / Linux (bash)

python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

Then run it from the project folder:

python -m gems_t4 scenarios
New terminal? Re-run the activate line. Prefer a shorter gems_t4 command instead of python -m gems_t4? Also run pip install -e ..

Quick start

python -m gems_t4 scenarios                          # list the fault scenarios
python -m gems_t4 live                               # live data, healthy vehicle
python -m gems_t4 live --scenario coolant_sensor     # live data with a fault injected
python -m gems_t4 dtc read --scenario misfire_cyl3   # read stored fault codes
python -m gems_t4 dtc clear --scenario misfire_cyl3  # clear stored fault codes
python -m gems_t4 actuator fuel_pump --state on      # run an actuator test

Everything runs against the virtual ECU by default. Add --port COM5 (Windows) or --port /dev/ttyACM0 (Linux) to talk to a real Pico adapter instead.

Graphical interface

A PySide6 kiosk GUI reproduces the Win98 dealer-tool look over the same engine: a boot splash, VIN-first vehicle identification (where you pick the fault scenario), a system menu, and screens for fault codes, live data (37 measures on painted analog gauges, with the authentic gauge-count → refresh-rate trade-off), actuator tests (refusals and all), toolbox self-tests, and a Programming sub-menu (gated coding editor, immobiliser Security-Learn re-sync, and the read-only fuel/ignition map viewer). Operations show the period-authentic "Communicating with ECU - please wait" progress overlay — click it to skip the wait, or launch with --instant to disable the waits entirely.

It needs the gui extra (PySide6). Install and launch:

pip install -e ".[gui]"
python -m gems_t4 gui                         # or: python -m gems_t4.app.gui
python -m gems_t4 gui --scenario coolant_sensor
python -m gems_t4 gui --instant               # no simulated ECU-communication waits

The CLI and the GUI are thin presentations over the same Backend/gems_core — no diagnostic logic is duplicated between them.

Commands

scenarios

List the available fault scenarios.

live [options]

Read live sensor data (the GEMS $61 records).

OptionMeaning
--scenario NAMEInject a fault scenario (default healthy).
--ids 0x01 0x02 …Read only these local ids instead of all of them.
--latency SECONDSModel authentic K-line slowness (default 0.0).
--port PORTUse a real Pico adapter on this serial port.

Common local ids: 0x01 coolant temp, 0x02 engine speed, 0x03 battery, 0x04 throttle, 0x05 MAF, 0x07 O2 bank A, 0x0A IACV steps, 0x0F fuel temp.

dtc {read|clear} [options]

Read or clear stored fault codes. Same --scenario / --port options as live.

actuator NAME [options]

Command an actuator output test.

NAMEActuator
milMalfunction indicator lamp
o2_heaterOxygen sensor heater
fuel_pumpFuel pump relay (refused while the engine “runs”)
ac_grantAir-conditioning grant
condenser_fanCondenser fan relay

Options: --state on|off (default on), plus --scenario / --port. Exit code is 0 if the test ran, 1 if the ECU refused it.

coding {read|write} — ECU coding (gated)

Read the coding block, or write one field through the safety gates (backup + verify + confirm). Writable identity fields (VIN last-6, dealer id, 4.0/4.6 select, transmission, build code) vs read-only factory fields (market, part number).

python -m gems_t4 coding read
python -m gems_t4 coding write --field vin_last6 --value 654321   # ASCII for vin/part, else hex

Writes ask for confirmation (Write VIN (last 6): '123456' -> '654321'? [y/N]) before touching the ECU; pass --yes/-y to skip the prompt in scripts.

immo {status|learn} — immobiliser Security-Learn

The one genuine GEMS over-the-wire write: re-sync the BeCM↔ECM mobilisation (the canon "ENGINE IMMOBILISED" recovery). --immobilised starts the virtual ECU desynced so you can watch the fix.

python -m gems_t4 immo status --immobilised
python -m gems_t4 immo learn  --immobilised

GEMS fuel/ignition maps are not writable over the K-line (bench EPROM swap) — the map viewer lives in the GUI as a read-only lookalike.

Fault scenarios

Each scenario produces coherent symptoms across fault codes, live data, and actuator tests, so a read → diagnose → clear workflow makes sense.

ScenarioCodeWhat you'll see
healthy—No faults; nominal live data; all tests allowed.
coolant_sensorP0118Coolant reads a fail-safe −40 °C; idle bumped high.
misfire_cyl3P0303Cylinder-3 misfire; rough/low idle.
lambda_heaterP1185O2 sensor heater open circuit; the o2_heater test is refused.

Example — the fault ties together across screens:

python -m gems_t4 dtc read --scenario lambda_heater            # -> P1185
python -m gems_t4 actuator o2_heater --scenario lambda_heater # -> REFUSED (same fault)

Real hardware (optional)

The canonical adapter is a Raspberry Pi Pico + ISO 9141 Click (L9637D) over USB. The Pico owns the K-line timing; the PC speaks a simple USB protocol to it.

  1. Build the parts list and wire it up per firmware/README.md.
  2. Flash firmware/pico_kline/pico_kline.ino to the Pico.
  3. Point the CLI at it:
python -m gems_t4 live --port COM5           # Windows
python -m gems_t4 live --port /dev/ttyACM0    # Linux

The host protocol between PC and Pico is specified in firmware/HOST_PROTOCOL.md.

Safety: OBD pin 16 is battery-live with the key off — fuse it. Start every session read-only; test any write (coding / Security-Learn) against the virtual ECU first.

Project layout

gems_t4/
  transport/   the only I/O layer — virtual ECU link, Pico adapter (USB), TCP (network), FTDI stub
  protocol/    KWP2000 framing, init, services, security, the KwpClient
  gems/        GEMS meaning — DTCs, $61 live data, actuators, immobiliser, maps, virtual ECU
  app/         Backend facade + Rich CLI + gui/ (PySide6 Win98 kiosk)
firmware/      Pico sketch + the USB-CDC host-protocol spec + wiring guide
tests/         pytest suite (runs with no hardware)
tests_regression/  independent regression suite from the frozen contracts (v0.0.4)
requirements.txt  runtime + test dependencies
INTERFACES.md  the frozen build contract (wire format, service map, module APIs)

The load-bearing seam is KwpClient(transport): it takes a real serial transport or the in-memory VirtualTransport, so the whole stack is testable off-car.

Development

Run the test suite (no hardware needed):

pytest                    # the original suite
pytest tests_regression   # the independent v0.0.4 regression suite (run explicitly)

Expected: 153 passed if you installed the gui extra (pip install -e ".[gui]"), or fewer passed with the GUI files skipped without it — the GUI test files need PySide6 and skip cleanly as whole modules (not an error) when it's absent, since requirements.txt deliberately doesn't include it (see "Graphical interface" above). tests_regression/ (new in v0.0.4) is a 234-test independent suite written from the frozen contracts — deliberately not derived from tests/ — and lives outside the default testpaths, so run it explicitly.

If you extend the code, keep to INTERFACES.md — it pins the wire format, the service/SID map, and every module's public API so the layers stay compatible. Guidelines: Python 3.11+, full type hints, docstrings, and no I/O or sleep in protocol/ or gems/ (timing lives only in transport/).

Use it from Python directly:

from gems_t4.gems.virtual_ecu import VirtualEcu
from gems_t4.gems.scenarios import get_scenario
from gems_t4.transport.virtual import VirtualTransport
from gems_t4.protocol.client import KwpClient
from gems_t4.gems import dtc, livedata

ecu = VirtualEcu(get_scenario("coolant_sensor"))
client = KwpClient(VirtualTransport(ecu))
client.connect()
client.start_session()

print([d.code for d in dtc.read_dtcs(client)])     # ['P0118']
print(livedata.read_all(client)[0].formatted())     # coolant temperature

Windows build

PyInstaller packaging (see packaging/README.md) produces a single dist/gems_t4/ folder containing two executables: gems_t4.exe (console — the CLI) and gems_t4_gui.exe (windowed, no console — the kiosk experience). Build with the [build] extra: pip install -e ".[build]".

Status

Phases 1–6 complete: the virtual ECU, full protocol stack, CLI, Pico firmware/transport, the PySide6 Win98 kiosk GUI (full period skin, painted gauges, the "Communicating with ECU" latency model), gated programming/coding/immobiliser/maps, 37 live-data parameters including per-cylinder misfire counts, and the two-exe Windows build — backed by a 153-test suite including an end-to-end GUI tour. The ECU link can also run over TCP (gems_t4 serve + --connect HOST[:PORT], or the GUI’s Configuration → VCI connection screen) — read-only over the network unless writes are explicitly enabled; see README.md “Remote / network use”. Remaining: on-car validation of the Pico adapter (needs real hardware). See the “Tech stack” and “Build status” sections of CLAUDE.md for the roadmap.
